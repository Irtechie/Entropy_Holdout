# KB: EB Remaining Single-Box Runs

## Origin

Source: `todo.md`

This plan executes the remaining non-GX10 EB targets one at a time after the small-model baseline at `runs/EB/EB-local-small-20260525-012128/`.

## Execution Rule

Each slice runs:

```powershell
pwsh .\tools\run_entropy_serial_experiment.ps1 -ExperimentPath .\benchmarks\entropy_workloads\experiment.local-full-singlebox.json -OnlySet <target> -ResultRoot .\runs\EB\EB-<target>-<timestamp>
```

After each run:

```powershell
pwsh .\tools\report_entropy_results.ps1 -ResultsPath <run>\results.jsonl -Json
git add <plan files/todo if status changed> <run folder>
git commit -m "data: add EB run for <target>"
git push origin main
```

If a target fails activation or the controller becomes unavailable, mark only that slice blocked and continue with the next runnable slice unless the failure indicates the whole controller is down.

## Failure Policy

Wave 2 is a baseline collection wave. Do not repair generated output inside a `plain` run.

Repairing LLMCommune/model serving is different from repairing model output. If activation, model loading, route state, cooldown, or controller status fails, the slice may repair or retry the serving path before classifying the result. That is infrastructure recovery, not benchmark intervention.

Treat these as completed EB data points:

- strict JSON/output-contract failures,
- missing expected artifact paths,
- validator failures,
- normal model refusals or malformed code,
- target-specific activation/model-serving failures that are diagnosed, captured, and do not leave the controller wedged.

Allowed mid-slice infrastructure recovery:

- wait out LLMCommune activation cooldowns,
- retry activation when the controller reports a transient state,
- inspect `/api/llm-host/current` and `/v1/models`,
- restart or reconcile the target only if the existing repo/controller scripts expose a safe non-destructive path,
- rerun the same EB slice after the serving path is healthy and record the recovery in the plan notes.

Not allowed in Wave 2:

- editing generated workload files to make validation pass,
- asking the model a repair prompt after a malformed answer,
- extracting files manually from malformed prose and counting that as `plain`,
- changing validators to make a run pass.

Treat these as blockers:

- controller unavailable across multiple targets,
- activation leaves the host in `reconcile_needed=true`,
- target load wedges the serving lane or prevents later targets from activating,
- run folder lacks `results.jsonl`, `critique.md`, or `critique.json`.

Wave 3 should add a separate named harness mode, such as `repair-extract`, that can repair JSON/extraction failures mid-stage. Its results must be stored in separate EB folders and compared against Wave 2 rather than overwriting the `plain` baseline.
